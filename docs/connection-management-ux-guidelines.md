# 连接管理 UX 设计规格说明

## 基线来源

本文档以鸿蒙端 `Index.ets` 作为主参考实现，以 `Viewer.ets` 作为补充参考。

- `Index.ets` 负责“连接管理首页”的主体验，是跨平台对齐的第一基线。
- `Viewer.ets` 只用于补充“进入会话后如何回访连接管理”的次级能力，不应反向定义首页结构。

这次对齐后，连接管理的核心语义应明确为：

- 它首先是“已保存连接库”的首页；
- 它的主任务是“查找、选择、维护并打开一个连接”；
- 它不是以快速表单为中心的单页连接器；
- 它与远程会话页是前后衔接关系，而不是同页混合关系。

## 适用范围

- 连接管理首页
- 已保存连接列表
- 搜索、筛选与清空搜索
- 新增连接入口
- 编辑连接弹层
- 删除确认
- 打开远程会话前的选择与操作体验
- 会话内的连接管理回访入口

## 不作为主基线的内容

- `Viewer.ets` 中的调试工具
- 会话中的画布操作控件
- 仅在会话页内出现的临时面板形态
- 旧版未挂到首页主路径上的快速连接表单

如果其他平台已经存在“快速连接表单”，可以保留为补充能力，但不能替代“连接库首页”这一主模型。

## 页面目标

- 让回访用户优先通过历史连接快速进入会话，而不是重复填写参数。
- 让首次用户能以最短路径完成“新增连接 -> 保存 -> 打开”。
- 让连接对象的查找、选择、打开、编辑、删除职责分层明确。
- 让页面在窄屏、宽屏、桌面和移动端都保持一致的信息架构。
- 让跨平台团队在实现差异存在时，仍能对齐同一套操作语义。
- 让误触直接进入会话、误删连接和误判当前选中对象的风险尽量降低。
- 让触控、键盘、鼠标、遥控器等输入方式都能走通同一条主路径。

## 核心原则

- 连接库优先：主页先展示“有什么可用连接”，不是先展示一组输入框。
- 查找优先于填写：搜索和排序优先帮助用户复用已有连接。
- 选择与打开分离：点选列表项只表示当前选中，不应直接触发连接。
- 打开动作显式：进入会话必须通过明确的“打开/连接”动作完成。
- 维护动作次级：编辑、删除应始终弱于“打开连接”。
- 破坏性操作需确认：删除前必须二次确认。
- 会话切换清晰：连接管理页负责启动会话，会话状态展示归属会话页。
- 布局随场景增强：窄屏保持单列，宽屏可以增强为双栏，但不能改变主路径语义。
- 焦点可感知：当页面支持键盘、鼠标或遥控器时，必须让用户清楚知道“当前焦点”和“当前选中”的区别与关系。

## 页面角色定义

### 1. 连接管理首页

- 这是跨平台统一的主页面角色。
- 承载连接库浏览、搜索、选择、打开、新增入口和关于入口。
- 用户进入该页后的首要判断应是“我是否已经有可直接打开的连接”。

### 2. 连接编辑弹层

- 用于新增或编辑单条连接配置。
- 与列表页解耦，避免在主页堆叠过多输入项。
- 适合容纳完整连接参数和低频配置。

### 3. 会话内回访入口

- 连接打开后，连接管理降级为会话内可回访能力。
- 它只负责回到同一连接对象的管理语义，不重新定义首页结构。
- 具体容器可以是抽屉、侧栏、浮层或系统面板。

## 信息架构

连接管理首页的信息架构固定为以下顺序，不建议跨平台打乱：

1. 页面头部
2. 搜索与新增区
3. 已保存连接列表或空态
4. 详情预览区（宽屏可选）
5. 行内主次操作
6. 弹层与确认反馈

原因如下：

- 用户先确认自己位于“连接管理”而不是“设置”；
- 然后通过搜索或滚动快速定位目标连接；
- 宽屏场景下，可以在不离开列表上下文的前提下查看当前选中对象详情；
- 再对目标连接执行“打开 / 编辑 / 删除”；
- 新增连接是入口动作，但不应压过历史连接复用；
- 会话页相关反馈不应回灌到首页主结构中。

## 主路径定义

### 回访用户主路径

1. 进入连接管理首页
2. 通过搜索或滚动定位连接
3. 选中目标连接
4. 点击“打开连接”进入会话页

### 首次用户主路径

1. 进入连接管理首页
2. 点击“新增连接”
3. 在弹层填写信息并保存
4. 新连接出现在列表前部并成为当前选中项
5. 点击“打开连接”进入会话页

### 重度用户快捷路径

1. 进入页面后直接使用键盘或遥控方向键移动选中项
2. 按回车打开当前选中项
3. 按删除键触发删除确认

这说明该页面不仅支持触控，也承担桌面和大屏场景下的键盘导航职责。

## 页面结构规格

### 头部区

- 标题固定为“连接管理”或平台等价表达。
- 右上角放置版本信息和“关于”入口。
- “关于”属于辅助入口，不应与“新增连接”竞争主路径。

### 搜索与新增区

- 搜索框位于列表上方。
- 搜索 placeholder 应明确说明检索范围，例如“搜索：名称 / 地址 / 账号”。
- 搜索命中字段至少包括：
  - 名称
  - 地址
  - 账号
- 当搜索框非空时，显示“清除”按钮。
- “新增连接”放在搜索区右侧，作为高可见但非唯一主操作。

### 已保存连接列表

- 列表按最近使用时间倒序排序。
- 列表项必须至少展示：
  - 名称；为空时回退为地址
  - 地址
  - 最近使用时间
- 允许选中态，且同一时刻只能有一个选中项。
- 选中态应通过背景、边框或字重联合表达，不能只靠颜色。
- 当页面支持焦点导航时，列表项还应有与“选中态”不同的“焦点态”。
- 列表项的主要信息应在 1 到 2 秒内可扫读完成，不应堆入低频配置字段。

### 宽屏详情预览区

- 仅在平板、桌面、大屏等横向空间充足时启用。
- 它是列表的补充层，不是替代层，不能把列表压缩成过窄导航栏。
- 详情区默认展示当前选中连接的摘要信息和快捷操作。
- 详情区可以复用“打开连接 / 编辑 / 删除”动作，但视觉主次关系必须与列表一致。
- 若列表与详情区同时存在，应避免出现两个同等抢眼的主按钮组。
- 当搜索结果中当前选中项被过滤掉时，详情区应明确提示“当前搜索结果中没有选中项”或等价表达。

### 行内操作

- 每条连接至少提供三类动作：
  - 打开连接
  - 编辑
  - 删除
- “打开连接”是行内主动作，视觉优先级最高。
- “编辑”是中性次级动作。
- “删除”是危险动作，必须弱于“打开连接”，并使用危险色。
- 点击整行只切换选中态，不直接打开连接。
- 宽屏若同时存在详情区操作与行内操作，两者必须语义一致，不能一个打开当前选中、另一个打开当前行却没有明确提示。

### 空态

- 当没有任何连接配置时，展示空态卡片。
- 空态主文案应说明当前没有可用连接。
- 空态必须提供明确动作“新增连接”。
- 空态不应展示无意义的占位骨架。

### 搜索无结果态

- 当已有连接但没有匹配结果时，展示“未找到匹配的连接”。
- 搜索无结果态与“完全无数据”必须区分。
- 搜索词应保留，便于用户立即修正。

### 编辑弹层

- 标题根据上下文切换为“新增连接”或“编辑连接”。
- 字段顺序固定为：
  - 名称（可选）
  - 地址
  - 账号（可选）
  - 密码（可选）
  - 记住密码
  - 触摸滚动步长
- 底部固定“取消 / 保存”双按钮布局。
- 点击遮罩或关闭按钮可以退出，但平台若支持未保存提醒，建议补充脏状态确认。

### 编辑弹层中的基础与高级分层

- 基础页只承载高频、首次用户也能快速理解的信息：
  - 名称
  - 地址
  - 账号
  - 密码
  - 记住密码
- SSH 相关能力统一收纳到高级页，不在基础页重复暴露。
- “记住密码”属于桌面连接本身的核心属性，应停留在基础页，而不是放到 SSH 或低频配置区。
- 不要在基础页加入“先完成基础连接”“更多选项”这类弱信息密度文案；如果字段本身已经足够自解释，应优先删除提示而不是继续堆文案。
- 高级页的目标是承载低频但完整的连接能力，不应与基础页形成重复输入。

### SSH 默认行为

- 新增连接时，SSH 默认关闭，避免把低频能力伪装成必填流程。
- 新增连接弹层必须从空白状态开始，不得带入上一次输入内容。
- 用户填写基础信息后，首次开启 SSH 时，允许系统按当前基础信息自动生成一套可工作的默认配置。
- 自动预填的默认规则应为：
  - `sshHost` 默认取桌面地址中的主机
  - `sshPort` 默认取 `22`
  - `sshUser` 默认复用基础页账号
  - `sshPassword` 默认复用基础页密码
  - `reuseDesktopUser` 与 `reuseDesktopPassword` 默认开启
  - `remoteHost` 默认取远端回环地址 `127.0.0.1`
  - `remotePort` 默认取桌面地址中的端口，未填写时回退到 VNC 默认端口
- 将 `remoteHost` 设为远端回环地址而不是 VNC 主机地址，是默认安全策略：
  - 优先假设 VNC 服务与 SSH 服务部署在同一台主机
  - 降低默认配置把 SSH 服务器再转发到其他内网主机的暴露面
  - 只有用户明确知道自己需要跳转到另一台内网目标时，才修改转发地址
- 当用户关闭 SSH 后再次开启时：
  - 若之前没有形成有效自定义 SSH 配置，可重新使用自动预填
  - 若用户已经修改过 SSH 明细，则不应被新的默认值强制覆盖
- 当 SSH 仍处于“重用基础账号密码”模式时，基础页账号或密码发生变化，SSH 对应字段应同步跟随，避免界面显示与实际生效值不一致。

## 状态模型

这里的状态模型以 `Index.ets` 首页职责为准，而不是把会话页状态混入首页。

### 初始加载

- 页面加载连接列表与上次选中项。
- 如果存在上次选中项，应恢复该上下文。
- 如果不存在上次选中项，则默认选中排序后的首项。
- 如果恢复的选中项已不存在，应回退到当前列表首项，而不是保留无效状态。

### 正常浏览

- 用户可搜索、选择、编辑、删除、打开连接。
- 列表保持稳定顺序，不因选中动作而重排。

### 无连接

- 页面呈现空态。
- 主推荐动作是“新增连接”。

### 搜索中

- 筛选仅影响显示集合，不改变底层列表顺序。
- 搜索为空时恢复完整列表。
- 搜索命中后，应尽量保留用户当前选中上下文。
- 如果当前选中项不在搜索结果中，可保留底层选中 ID，但显示层必须提供清晰反馈，避免误以为“没有选中”。

### 已选中

- 当前连接项获得稳定选中态。
- 键盘操作默认作用于当前选中项。
- 若平台存在详情预览区，它应始终绑定到当前选中项，而不是绑定到悬停项或焦点项。

### 焦点浏览中

- 当页面处于键盘、鼠标或遥控器输入模式时，应提供可感知的焦点位置。
- 焦点可以落在搜索框、新增按钮、列表区、行内操作或详情区操作上。
- 焦点顺序应符合视觉顺序：头部辅助入口 -> 搜索与新增 -> 列表 -> 详情区动作 -> 弹层。
- 列表获得焦点后，方向键默认作用于列表选择，而不是页面整体滚动。

### 编辑中

- 弹层置顶。
- 背景列表保留上下文，但不可作为主交互层。

### 删除确认

- 删除前必须弹出确认对话。
- 确认文案需带连接名称或地址，降低误删风险。

### 打开中

- 首页职责是“发起进入会话”。
- 一旦打开会话页，连接状态与错误反馈应交由会话页处理。
- 首页不应同时承担“连接中 / 已连接 / 断开连接”的主显示职责。

## 关键交互规则

### 选择规则

- 点击列表项只代表“选中此连接”。
- 选中动作不等于连接动作。
- 这种分离必须跨平台保持一致，避免误触直接进入远程会话。
- 行选择后，当前对象的详情、快捷键和辅助操作都应同步切换到该对象。

### 打开规则

- 打开连接必须通过显式动作触发：
  - 行内主按钮
  - 键盘回车
  - 平台等价的确认手势
- 不建议把整行点击直接绑定为打开动作。

### 搜索规则

- 搜索是“定位器”，不是过滤配置页。
- 搜索结果中仍应保留行级操作，不应迫使用户先进入详情页再执行打开。
- 搜索输入过程中，不应因为筛选结果变化而突然触发打开、编辑或删除。
- 搜索框获得焦点时，方向键优先服务文本编辑；列表获得焦点时，方向键优先服务列表导航。

### 排序规则

- 列表默认按最近使用倒序。
- 选中不会更新时间。
- 进入会话时可以刷新“最近使用时间”。
- 如果平台能拿到真实连接成功结果，推荐额外记录“最近成功连接时间”，但不能破坏默认排序心智。

### 删除回退规则

- 删除非选中项时，当前选中保持不变。
- 删除当前选中项时，自动回退到新的首项。
- 若列表已空，则回到空态。
- 删除确认弹层关闭后，焦点应回到触发删除的来源区域，避免键盘用户丢失上下文。

### 名称策略

- 名称允许为空。
- 名称为空时，主标题回退显示地址。
- 如果会话页能返回远端设备名，且用户未手工命名，可自动补齐名称。
- 自动补齐不能覆盖用户已有命名。

### 键盘与大屏规则

- 支持上下方向键切换选中项。
- 支持回车打开选中项。
- 支持删除键删除选中项。
- 这套规则对桌面端、TV、大屏设备应保持一致。
- 当列表可滚动时，切换选中项应尽量自动滚动到可见区域。
- 键盘快捷路径不应跳过删除确认。

### 鼠标与悬停规则

- 悬停可以提供弱提示，但不能替代稳定选中态。
- 右键菜单、悬浮工具条等增强能力可以存在，但不应成为唯一操作入口。
- 鼠标双击若被平台视为高频打开手势，必须仅作为可选增强，不能替代单击选中 + 显式打开这一基础语义。

## 视觉与布局规格

### 整体布局

- 首页应采用“完整连接库页面”布局，而不是局部悬浮卡片。
- 页面主体建议纵向结构：头部 -> 搜索区 -> 列表区。
- 列表是页面视觉主体。

### 容器宽度

- 窄屏使用单列全宽内容区。
- 宽屏可使用受控内容宽度，但不要把列表压缩成过窄侧栏。
- 若平台支持双栏，可扩展为：
  - 左侧连接列表
  - 右侧详情预览或操作区
- 但“搜索在上、列表为主、打开为主动作”的语义不能变。
- 双栏模式下，建议列表区承担主要浏览任务，详情区承担解释、确认和快捷操作任务。
- 双栏模式应避免让详情区比列表区更抢眼，防止页面重心从“找连接”偏成“看详情”。

### 间距系统

- 采用 4/8 节奏。
- 页面左右内边距建议 `16` 起。
- 列表项内边距建议 `14-16`。
- 搜索区和列表区之间保持清晰分组。

### 圆角与层级

- 列表项可使用中大圆角卡片化样式。
- 选中项需要额外边框或底色提升层级。
- 弹层层级始终高于列表和辅助弹窗。
- 焦点态若存在，应再高于普通列表项，但不应压过模态弹层。

### 字体层级

- 页面主标题建议 `18`
- 搜索与列表主信息建议 `14-15`
- 地址和时间等次信息建议 `12`
- 辅助入口建议 `12`

### 触达尺寸

- 图标按钮最小点击区域不低于 `42x42`
- 触控平台推荐不低于 `44x44`
- 搜索框高度建议不低于 `40`
- 弹层底部按钮高度建议不低于 `36`

### 动作密度

- 同一条连接上的主次操作不宜超过 3 个常驻按钮。
- 若宽屏同时提供详情区动作，列表内动作可以更紧凑，但不能完全隐藏“打开连接”主动作。
- 删除动作在视觉上应与编辑动作区分开，避免连续误触。

## 内容文案规格

### 推荐命名

- 连接管理
- 新增连接
- 编辑连接
- 删除连接
- 未找到匹配的连接
- 暂无连接配置

### 列表文案

- 最近使用时间应使用用户可理解表达，例如“上次连接 2026-04-03 14:20”。
- 删除确认中优先展示名称，没有名称时展示地址。

### 表单文案

- 地址字段必须给出明确示例，例如 `192.168.1.2:5900`。
- 可选字段在标签中标注“可选”。
- 低频能力不要只靠 placeholder 解释。

## 响应式与跨场景要求

### 手机

- 保持单列结构。
- 搜索、清除、新增都必须在首屏可见。

### 平板与桌面

- 可以增强列表密度和键盘支持。
- 可以引入 hover、右键或快捷键，但不能替代显式主按钮。
- 推荐在宽屏下提供稳定的详情预览区，而不是简单把单列列表横向拉宽。
- 推荐提供键盘初始焦点策略，例如默认落在列表或搜索框，但要保持跨版本一致。

### 会话内回访

- 进入会话后，连接管理应降级为回访入口。
- 回访内容可以复用同一连接对象信息，但不应倒逼首页变成会话页的附属面板。

## 可访问性要求

- 搜索框和表单字段必须有可感知标签。
- 选中态、危险态、主按钮态不能只靠颜色表达。
- 行内多个按钮需要明确可访问名称，例如“打开连接”“编辑连接”“删除连接”。
- 键盘焦点顺序应与视觉顺序一致。
- 删除确认必须提供清晰的取消路径。
- 若同屏存在列表与详情区，屏幕阅读器应能区分“连接列表”和“当前选中详情”两个语义区域。
- 焦点回退、弹层关闭和删除后回退选中逻辑，对辅助技术用户必须可预期。

## 交互禁忌

- 不要把首页主模型定义成“表单优先”。
- 不要点击整行就直接打开远程会话。
- 不要把编辑和删除做成比“打开连接”更抢眼的动作。
- 不要把“无结果”和“无数据”混成同一种空态。
- 不要只在会话页里才允许管理连接，导致用户必须先进入一条连接才能修改配置。
- 不要把首页和会话内面板设计成两套完全不同的连接对象语义。

## 跨平台对齐要求

- 必须保持“连接库首页”作为主入口模型。
- 必须保持“搜索定位 + 列表选择 + 显式打开”的主路径。
- 必须保持“新增/编辑走弹层或独立表单，且弱于打开连接”的关系。
- 必须保持“删除需要确认”的风险控制。
- 必须保持“最近使用倒序”的默认排序逻辑。
- 必须保持“行点击只选中，不直接打开”的操作语义。
- 可以有平台差异的部分仅限于：
  - 容器样式
  - 图标与视觉语言
  - 原生对话框、抽屉、侧栏或弹层形态
  - 键盘、鼠标、遥控器、触控的具体交互映射
  - 是否启用宽屏详情预览区

## 平台落地建议

### 安卓与鸿蒙

- 优先保证触控主路径顺畅，再增强键盘和大屏体验。
- 若采用宽屏双栏，应让右侧详情区成为“确认当前对象”的辅助层，而不是另一个表单区。
- 若同屏存在行内按钮和详情区按钮，应统一文案、危险色和禁用态规则。

### 桌面端

- 优先把列表焦点、键盘导航、自动滚动可见区和删除后焦点回退做完整。
- 可以增强快捷键、右键菜单和 hover，但不能破坏基础显式动作模型。

### iOS 与 Web

- iOS 应优先遵守触控与分层原则，不必为了桌面一致性强行引入复杂 hover。
- Web 若支持鼠标双击打开，仍必须保留单击选中与显式打开按钮。

## 实现约束

- 每条连接必须有稳定 ID，不能依赖列表索引。
- 选中连接 ID 需要持久化，用于恢复上下文。
- 搜索命中逻辑需要统一覆盖名称、地址、账号。
- 删除当前选中项后要自动处理回退选中逻辑。
- 最近使用时间需要与“打开会话”动作关联，而不是与“选中”动作关联。
- 如果平台未来区分“发起打开”和“连接成功”，建议同时保留更精细的时间字段，但对外仍维持统一排序语义。

## 建议的验收清单

- 是否能在 5 到 10 秒内理解这是“连接库首页”而不是设置页？
- 是否能在不新增输入的情况下，从历史连接中快速打开目标设备？
- 是否能清楚区分“选中一条连接”和“真正打开连接”？
- 是否能通过搜索快速按名称、地址或账号定位连接？
- 是否能在空态下明确知道下一步是“新增连接”？
- 是否能在搜索无结果时快速恢复列表？
- 是否能在桌面或大屏场景中用键盘完成选择和打开？
- 是否能在进入会话后仍回访该连接对象，而不是丢失管理上下文？
- 是否能在宽屏场景下同时看清“当前选中对象是谁”和“我接下来可以做什么”？
- 是否能在删除、搜索过滤、弹层关闭后保持稳定焦点与选中回退？

## 建议的体验指标

- 回访用户从进入页面到打开目标连接，理想应控制在 `3` 次主操作以内。
- 首次用户从进入页面到完成“新增并打开”，理想应控制在 `1` 次入口点击加 `1` 次保存加 `1` 次打开。
- 搜索命中后的目标连接，应尽量在首屏或一次短滚动内可见。
- 用户在窄屏和宽屏之间切换时，不应重新学习“如何打开一个连接”。

## 演进守则

- 后续新增能力应优先判断它属于“连接库首页”还是“会话页”，避免职责污染。
- 若未来加入标签、分组、收藏、最近成功连接等能力，必须继续服务“快速定位并打开”这一主任务。
- 若未来加入导入导出、批量操作、云同步等能力，应作为次级入口，不应压过搜索和打开主路径。
- 文档中的平台差异允许扩展视觉和交互映射，但不得回退到“表单优先”或“整行直接打开”的旧模型。

## 近期迭代记录

### 2026-04 连接编辑与 SSH 体验收敛

- 本轮对齐覆盖 Android 与鸿蒙，两端以同一套编辑语义为准，不再允许一端保留旧交互、一端切到新交互。
- 已确认并固化以下设计决策：
  - “记住密码”放回基础页
  - SSH 主开关与 SSH 明细统一放到高级页
  - Android 侧去除重复的“保存密码”入口
  - 合并 SSH 账号/密码复用开关，默认按基础账号密码整体重用
  - 删除“更多选项”“先完成基础连接”等低价值提示文案
  - 修复 SSH 开关关闭后无法再次开启的问题
  - 新增连接不再继承上一次编辑内容
  - 新增连接默认不启用 SSH
  - 首次启用 SSH 时，按基础信息自动补全一套默认可用配置
  - SSH 默认转发目标改为远端 `127.0.0.1`，不再默认指向桌面主机地址
- 这轮迭代的目标不是增加配置能力，而是把“首次用户可直觉完成新增，高阶用户再逐步展开 SSH 明细”作为统一 UX 主线。
- 后续若继续优化连接编辑体验，应优先检查：
  - 是否仍保持基础页高频、SSH 低频的分层
  - 是否仍保持新增连接为空白起点
  - 是否仍保持 SSH 默认值安全且可解释
  - 是否仍保持 Android 与鸿蒙操作语义一致

## 结论

基于鸿蒙 `Index.ets` 提炼后的连接管理主模型，不是“快速连接表单页”，而是“连接库首页”。

它的核心价值在于：

- 先复用，再填写；
- 先定位，再打开；
- 把维护动作收纳为次级能力；
- 把会话状态留给会话页；
- 用一致的对象模型，为 Android、桌面端、iOS 和 Web 的后续改造提供统一基线。
